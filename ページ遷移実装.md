このリポジトリ内の全ページ共通で、内部ページ遷移時にローディング演出を出す仕組みを実装してください。

【前提】
- 現在の遷移用のJSは破棄
- 共通ヘッダー/フッターは、js/common-navigation.js から生成される構成です
- 既存の共通CSSは css/common.css
- 既存の共通JSは js/common-navigation.js
- 可能な限り、個別ページごとの大量修正ではなく、css/common.css と js/common-navigation.js を中心に全ページへ反映すること
- 擬似デモではなく、実際のHTMLページ遷移として動く実装にすること
- 専用の中継ローディングページは作らないこと
- 遷移は必ず「遷移元ページ → 遷移先ページ」の1回だけにすること

【対象ファイル】
- 共通CSS: css/common.css
- 共通JS: js/common-navigation.js
- 代表ページ: index.html
- 共通でheadに入っているテンプレートまたは親HTML: NONE

【やってほしいこと】
1. js/common-navigation.js を、共通ヘッダー/フッター生成だけでなく、全ページ共通のページ遷移演出オーケストレーターとして拡張する
2. js/common-navigation.js から、ローディングオーバーレイ用DOMを全ページに共通で差し込む
3. css/common.css に演出用スタイルを追加する
4. js/common-navigation.js で内部リンク遷移を横取りして前半演出を開始する
5. 遷移先ページでは sessionStorage を使って後半演出を再開する
6. 元ページや素の遷移先ページが一瞬でも見える状態を防ぐ
7. この仕組みを、同一リポジトリ / 同一サイト内の内部遷移すべてに適用する
8. 外部リンク、target=\"_blank\"、mailto:、tel:、ページ内アンカー、ダウンロードリンクなどは除外する

【最重要要件】
- 現状の問題は、ローディング演出の途中や遷移直後に、見せたくない元ページや素の遷移先ページが見えてしまうこと
- これを必ず防ぐこと
- 特に遷移先ページの初回描画時に、素のページ本体が見えないようにすること
- そのため、head のかなり早い段階で sessionStorage を見て html 要素に再開用クラスを付ける方式を優先すること
- もし共通で head に入るテンプレートや親HTMLがあるなら、そこに最小限の inline script を追加すること
- もし共通の head テンプレートが存在しないなら、チラ見え防止のために必要な最小変更をどこに入れるべきか判断し、代表ページだけでなく全体に適用できる方法を選ぶこと
- js/common-navigation.js だけでは初回描画前制御が間に合わない場合、その事実を踏まえて最小限の head 側修正を行うこと

【演出仕様】
演出全体は約3秒とする

進捗は以下の段階でよい
- 0%
- 16%
- 44%
- 72%
- 99%

表示ルールは以下を厳守する
- 0%〜16%: 遷移元ページが徐々に薄くなっていく
- 16%〜72%: ページ画像は見せない。ローディング演出のみを見せる
- 72%〜99%: 遷移先ページが徐々に濃くなっていく
- 最後にオーバーレイが消えて、遷移先ページが完全表示される

【背景演出の確定仕様】
見た目は以下の方向性で実装すること

1. 2進数
- 0 と 1 の数字を使う
- 上下から中央の消失点へ向かって流れ込む
- 中央奥へ吸い込まれる遠近感を出す
- 進むほど小さくなり、中央で消えるようにする

2. 16進数
- 16進数コードを左右から中央の消失点へ流し込む
- 単なる横流れではなく、中央奥へ吸い込まれる遠近感を出す
- 進むほど縮小し、中央で消えるようにする

3. 64進数
- 64進数風の短いコード列を追加する
- 時計回りの渦を描きながら中央の消失点へ吸い込まれるようにする
- 存在感ははっきり出すが、2進数と16進数を邪魔しないようにする

4. ニューロン / シナプス内部イメージ
- 外周部に、ニューロンのシナプス内部のような表現を入れる
- シナプス終末の膨らみ、小胞、枝状の接続線、中心へ流れる電気信号のパルスを入れる
- 電気信号のパルスは視認しやすくする
- 周辺から中心へ「信号が走る」感じを出す

5. 中央の消失点
- 進捗バーの少し上に中央の消失点を置く
- 画面情報がすべてその一点の奥へ吸い込まれる構図にする
- グローやリングなどを置いて、奥行きの核として分かるようにする

6. 将来のキャラクター用レーン
- 今回はキャラクター自体は実装しない
- 将来的に、アニメキャラクター風の女性が中央奥から手前へ小走りで来る予定
- そのため、中央上部から進捗バー上部へ向かう導線が読み取れる構図を壊さないこと

【UI仕様】
- オーバーレイ上にローディングパネルを置く
- 位置は下中央寄り
- パネルには以下を表示する
  - タイトル
  - 状態メッセージ
  - パーセンテージ
  - ローディングバー
  - 2進数 / 16進数 / 64進数が混ざった短い更新文字列
- ローディングバーはCSSで実装する
- 見た目はサイバー感、発光感、デジタル空間感を出す

【実装ルール】
- HTML / CSS / JavaScript のみで実装する
- 外部ライブラリは使わない
- 共通処理は css/common.css と js/common-navigation.js に集約する
- js/common-navigation.js は以下の責務を持ってよい
  - 共通ヘッダー/フッター生成
  - ローディングオーバーレイDOM生成
  - 内部リンク遷移フック
  - sessionStorage への保存
  - 遷移先ページでの演出再開
- ただし、初回描画前のクラス付与だけは必要なら head 側に最小限の inline script として分離する
- sessionStorage に以下のような情報を保存してよい
  - 演出開始時刻
  - 遷移状態
  - 遷移先URL
  - 演出再開フラグ

【内部リンク制御】
- js/common-navigation.js で内部リンククリックを検出する
- 同一オリジンかつ通常の内部遷移のみを対象にする
- 条件に合うリンクは、いったんデフォルト遷移を止めて前半演出を開始する
- 遷移元で 16% まで現在ページを薄く見せる
- 16%〜72% はページ画像を見せないので、この区間の適切なタイミングで遷移先へ移動してよい
- 遷移先では sessionStorage を読んで後半演出を再開する
- ブラウザ戻る/進むに対しても破綻しないよう、少なくとも異常状態でオーバーレイが残り続けない設計にする

【初回描画チラ見え防止】
- 最重要
- 遷移先ページでは、素のページ本体が一瞬でも見えないようにする
- head の早い段階で sessionStorage を見て html にたとえば transition-resume のようなクラスを付けることを優先する
- そのクラスがある間は、ページ本体を最初は見せず、ローディングオーバーレイを前面に出すこと
- 72% まではページ本体を見せないこと
- 72%〜99% でページ本体を徐々に見せること
- js/common-navigation.js の実行タイミングだけでは間に合わない場合は、その理由を踏まえて最小限の head 修正を行うこと

【common-navigation.js が生成元であることへの配慮】
- 既存のヘッダー/フッター生成処理を壊さないこと
- 既存のナビゲーション生成ロジックと競合しないように責務を整理すること
- 可能なら以下のように構造化すること
  - 共通DOM生成
  - 遷移演出初期化
  - 内部リンクフック
  - 遷移再開処理
- 既存コードが大きい場合は、関数分割して可読性を上げること
- 既存のイベント登録やリンク処理を不用意に壊さないこと

【実装の進め方】
1. まず css/common.css、js/common-navigation.js、代表ページ、必要なら共通head注入元を確認する
2. どこに以下を入れるのが最小変更で済むか判断する
  - head の早期再開判定
  - オーバーレイDOM
  - 共通スタイル
  - 共通ロジック
3. その方針を短く示す
4. 実装する
5. 変更したファイル一覧を最後に示す
6. 元ページや素の遷移先ページのチラ見えをどう防いだかを短く説明する
